the lambda calculus⌗

something that i have noticed about the lambda calculus is the fact that it is very simple given a programming background; but is full of strange and mysterious names for simple concepts that we programmers see in everyday life.

i create these notes as a reference to some of the notations and concepts present within the lambda calculus, as well as some supplementary material such as theorems and definitions.

lambda calculus is a formal system of logic which aims to describe computation. we can prove that the system is indeed turing complete via the church-turing thesis, which states that; any problem which is computable by the lambda calculus.

the relation between the lambda calculus and turing machine can be formally defined via the church turing thesis, which states what any problem which can be effectively calculable in finite time by a turing machine, can be described via the lambda calculus. (i am sort of paraphrasing, so apologies if it is not as correct as the original definition of the thesis).

we know this term simply as turing completeness, and its a trait that is demonstrated within the majority, if not all programming languages shown today. so by definition, the lambda calculus is technically a bare minimum for a programming language. (with the exception of smaller formal systems such as iota/jot/zot).

basic syntax⌗

the basic syntax of the lambda calculus is as follows

<term> -> <variable>
      | λ <variable>. <variable>
      | ( <term> <term> )

the syntax for the system of logic is so incredibly simple, but as we will see later; it holds the computational power equivalent to that of the turing machine (via the church-turing thesis).

before we proceed, something that i should inform you of is the notation that i will be using throughout the rest of the post. i am not going to be strictly sticking with the syntax, and there will be slight deviations as i will mention now.

currying⌗

as we saw from the previous bnf definition of the syntax of lambda calculus, every application must only contain a single variable. this means that we cannot pass more than one parameter to a function/ abstraction within the lambda calculus.

for example, here is an invalid function abstraction which performs addition.

\xy.(x + y)

lets curry this function and correct it.

\x.(\y.(x + y))

very nice, now we have a valid abstraction.

free and bound variables | closure & combinators⌗

a bound variable is a variable which is declared within the current function scope.

on the other hand, a free variable is a variable which utilizes a variable from a parent scope. within programming, this can be thought of as using a global variable, or a lambda capturing a variable from a parent scope. in this case, the function can be thought of as a closure.

this may seem rather arbitrary, but some rather interesting complications arise as we begin to work with higher order functions. if we can have closures which inherit scope from a parent, what happens when we return this function? the closure is no longer present underneath the parent scope, so does the scope go with the closure, or does it simply cause an error?

within a lexically scoped language, the closure will keep its scope, and it’s free variables. this may differ in some languages, but it remains true in the lambda calculus.

a combinator on the other hand is the complete opposite, in the sense that it does not use any free variable. this is the only requirement of a combinator. the concept of a combinator can be further elaborated on within the study of combinatory logic (not to be confused with combinatorics).

there is going to be a section for combinatory theory at a later point within the blog. the only thing we need to understand at this point is the concept of the combinator/closure, and how they relate to free/bound variables and higher order functions.

beta reduction⌗

the concept of beta reduction is simple. we simply apply a variable to an abstraction, which we can do simply by substituting. this is simply a fancy name for a very very well understood concept within the rudimentary algebras.

here is a small demonstration of the beta reduction.

(\x.x) 3

we can reduce this simply by substituting the redex by 3, which results in 3.

3

now that was a fairly simple example of beta reduction, lets amp this up a bit and write a lambda abstraction which simply performs addition on the natural numbers.

(\x.(\y.(x + y)) 10) 20

now lets apply a beta reduction to this step by step and view the result.

1. (\x.(\y.(x + y)) 10) 20
2. (\y.(20 + y)) 10
3. (20 + 10)
4. 30

now as we can see, beta reduction simply means substituting the variables within the abstractions.

the church-rosser theorem⌗

when applying variables to lambda abstractions, we also need to know the order in which we are reducing/evaluating the abstraction.

the church rosser theorem states that, “when applying reduction rules to terms, the ordering in which the reductions are chosen does not make a difference to the eventual result”. this theorem essentially states that the lambda calculus is confluent.

so if the result of the abstraction is always the same, then why is the evaluation order/strategy significant?

the church rosser theorem states that all abstractions will always evaluate to the same result regardless of the strategy, but something to note is the fact that not all evaluation strategies will result in a final outcome.

so the theorem states that there is no logical fallacy within the lambda calculus, as it is confluent and will produce the same result after evaluation, but not all evaluation strategies will produce a correct result.

this is where the importance of reduction strategies lie, as we must use the correct order if we want to produce a correct final result.

here is a diagram of the proof for the theorem. this is much easier to visualize as opposed to the textual proof.

this proof of the church rosser theorem directly translates to, “if there are two distinct reductions or sequences of reductions that can be applied to the same term, then there exists a term that is reachable from both results, by applying (possibly empty) sequences of additional reductions”.

combinatory calculi⌗

combinatory calculi is a branch of mathematical logic which is rooted in the lambda calculus, but extends the system by eliminating the need for free variables. instead, it relies completely on the composition of a finite set of primitive combinatory terms. as we have previously defined, a combinatory term is simply a function that contains no free variables.

static typing⌗

the concept of static typing, as previously stated, is the act of the compiler inferring and handling the type checking at compile time; in the case of programming languages, the frontend compiler.

language such haskell, C, C++, java, rust and more use this; and is commonly seen within languages which compile to machine code, as opposed to some intermediate representation which acts as bytecode for some abstract runtime environment.

the following is a list of the benefits that i can currently think of for statically typed languages.

  • faster runtime spees as the environment does not have to worry about type checking.
    • this remains true for both interpreted and compiled languages.
  • a must for compiling to machine code, as hardware does not support type checking.
  • it makes bugs easy to catch early on, as the compiler can warn the user and terminate compilation.
  • there are probably much more, but my brain is fuzzy right now. sorry uwu.

dynamic typing⌗

a dynamically typed language is a language which performs type checking at runtime. this approach to type checking is only present within interpreted languages, since as previously stated; hardware has no support for type checking within its runtime. it only understand data and code, information and opcodes.

now before we get into the pros of dynamic typing, i want to first explain an extremely important concept within dynamic typing that almost all runtime environment use; and that is call ducktyping.

the concept of ducktyping is very simple. there is a funny little joke that is used that encapsulates this concept fairly well, and serves as the origin of the name: “if it walks like a duck, and quacks like a duck, then it must be a duck”.

this may seem a bit strange and arbitrarily random at first glance, but i will attempt to explain the meaning behind this statement. lets first start off with an example.

within python, everything is stored as an object at runtime. these include literally everything, from integers, lists, strings, functions, classes, modules; everything that is a tangible entity at runtime will be stored as an object. these objects themselves hold their own type information. this allows the object’s type to be changed at runtime dynamically simply by assigning it a different type and values.

when the python runtime want to evaluate something as simple as an expression, it will first need to ensure that the types within the expression are capable of arithmetic. it does do this by categorizing every single type into what it can and cant do, that would be way too inefficient for everybodies standards.

instead, it will simply check to see if the object is capable of the arithmetic operation. it doesnt care about its type at all, it simply check if the object is capable of the operation it wants to do. if the object looks like an integer that can perform arithmetic, then the interpreter will attempt to perform arithemtic on the object.

this drastically speeds up the the type checking, and by product the runtime speeds. i will explain the entire process of the type checking system at a later section within the post, if you are interested then you can skip there directly. there are no additional prerequisite materials for that section of the post besides a fairly general understanding of the c programming language.

the benefits of dynamically typed languages are as shown.

  • lack of type checking pass makes compilation speeds faster.
    • this is generally a massive benefit for scripting languages such as python and lua.
  • dynamic types allow for much faster object oriented runtime polymorphism.
  • generic programming does not have to be restricted at compile time.

strongly typed⌗

now we move onto the concept of strongly and weakly typed systems. something that is very important to note is the fact that these two have NO RELATION TO STATIC/DYNAMIC TYPING!!!! this is something that had confused me when i had originally come across these concepts.

the concept of strongly typed languages simply means that the language itself enforces a restriction on converting/performing operations between types. in this case, it simply means that an integer cannot perform an operation with a float if they are defined as two different types. languages such as haskell and C++ are strongly typed.

this may seem like a very odd and restrictive decision to emplace on a type system for a programming language, but it does come with it’s benefits and solutions.

for example, strongly typed languages will generally include something within their langauge known as ad-hoc or bounded polymorphism. this is when the user may explicitly state what function can operate on which types. i understand that this definition of ad-hoc polymorphism may be a bit vague, but i will do my best to explain this to the best of my capacity.

as you may know, C++ is not exactly the best example to show strong typing, as its foundations are built entirely off of the C programming language, which contains a concept known as implicit conversion. this allows for explicit type casting, and automatic conversions within expressions for a much more streamlined programming experience. for this reason, i will be using haskell to explain this concept.

haskell is a purely functional, statically and strongly typed language which is largely rooted in mathematics. this makes it a very beautiful language, but also a pain to code in. some people may disagree with this statement, but i believe that it is an irrefutable fact that there are better languages that are specifically tailored to the programming experience; as opposed to the purity of the language itself. an example of this is ocaml, but that another rant that i dont want to get into here.

within haskell, there is a concept known as a typeclass. this is a feature that allows you to define a finite set of methods that can operate across multiple types. if you don’t have a background in any functional language, i would highly recommend learning one. it will greatly aid in your understanding of the concepts present within the lambda calculus, which functional programming is largely based on. (i say largely, as some functional languages are not pure).

for those who have experience in C++, ad-hoc/bounded polymorphism is the same thing as operator overloading. operator overloading allows the user to overload functionality of methods/functions on specific types. this is an important staple within generic programming in C++.

the following is a list of the benefits of a strongly typed system.

  • type correctness within the program
    • if the programmer is forced to explicitly defined the set of operations which are allowed on a set of types, then the programmer will know that the expressions are indeed using the correct types; as opposed to allowing the compiler to perform implicit type casts.
  • static analysis is much more useful within a strongly typed languages as the analyzer knows each type of each operand within an expression.
  • eliminates an entire class of bugs/vulnerabilities that are present within weakly typed language, such as type confusion and insecurity.

weakly typed⌗

a weakly typed language is the opposite of a strongly typed language (wow really?), in the sense that the language does not enforce operations between exact types; and may allow for explicit conversion between types.

an example of a weakly typed language is the C programming language (semi-weakly typed). i chose C for this example, as an extremely well known concept within the language is type casting and implicit conversion (as we covered previously). this allows the programmer to explicitly cast the types present within an expression to satisfy the compiler.

this approach places much more freedom within the hands of the programmer and reduces the amount of code they have to write for semi-generic code, but also introduces a class of vulnerability into the language. if used incorrectly, the type safety of the program may potentially be compromised, and may cause undefined behaviour in the best case, and an exploitable bug in the worst.

however, weakly typed languages are not without benefits, as with all tradeoffs. the following is a list of the benefits of weak typing.

  • places much more control in the hands of the developer and allows them to explicitly convert types.
    • this largely depends on the language specification and compiler.
  • reduces code bloat and boiler-plate.
    • the user does not have to explicitly define the operations to be performed on each type to achieve ad-hoc/bounded polymorphism
